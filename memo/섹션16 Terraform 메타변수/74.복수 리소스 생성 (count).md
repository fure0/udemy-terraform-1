# 74. 복수 리소스 생성 (`count`)

**같은 `resource` 블록을 여러 번** 적용해, 비슷한 리소스를 한꺼번에 만드는 방법이다.  
섹션16에서는 **`count`** 와 **`for_each`** 두 가지가 있고, 이번 강의는 **`count`** 중심이다.

이전: **73 `depends_on`** · 다음: **75 `for_each`**

---

## 복수 리소스를 만드는 방법 (슬라이드)

| 종류 | 설명 |
|------|------|
| **`count`** | **지정한 횟수**만큼 리소스를 복제 |
| **`for_each`** | **리스트·맵**을 돌면서 리소스를 복제 (75번) |

**주의 (슬라이드):** **`count`와 `for_each`는 동시에 지정할 수 없다.**  
한 `resource` 블록에는 둘 중 하나만 쓴다.

---

## `count`란

**지정한 수만큼** 그 리소스 인스턴스를 생성한다.

```hcl
resource "aws_instance" "server" {
  count = 4

  ami           = "ami-01748a72bed07727c"
  instance_type = "t2.micro"

  tags = {
    Name = "Amazon Linux 2 Server ${count.index}"
  }
}
```

| 항목 | 의미 |
|------|------|
| `count = 4` | 이 블록으로 **EC2 4대** 생성 |
| `count.index` | **0부터 시작**하는 인덱스 (0, 1, 2, 3). 이름·태그 등에 붙여 **대별로 구분** |

`count`는 **숫자**다. 변수로 줄 수 있다.

```hcl
variable "server_count" {
  type    = number
  default = 2
}

resource "aws_instance" "server" {
  count = var.server_count
  ...
}
```

---

## `count`를 쓰면 달라지는 것

### 1. 리소스 주소

`count` 없을 때:

```hcl
aws_instance.server
```

`count` 있을 때 — **인덱스가 붙는다**:

```hcl
aws_instance.server[0]
aws_instance.server[1]
aws_instance.server[2]
aws_instance.server[3]
```

다른 리소스에서 참조할 때도 **`[인덱스]`** 가 필요하다.

```hcl
output "first_server_ip" {
  value = aws_instance.server[0].public_ip
}
```

### 2. 블록 안에서의 `count.index`

각 복제본마다 **같은 HCL**이 실행되며, `count.index`만 0, 1, 2… 로 바뀐다.

### 3. `count = 0`

리소스를 **하나도 만들지 않는다** (기능 플래그·환경별 on/off에 자주 사용).

```hcl
count = var.create_servers ? 1 : 0
```

---

## 여러 개 참조하기

**전체 목록**이 필요하면 **splat** `*` 를 쓴다.

```hcl
# 모든 인스턴스의 private IP 목록
output "all_private_ips" {
  value = aws_instance.server[*].private_ip
}
```

개수는 `length(aws_instance.server)` 로 다룬다.

---

## 이 프로젝트와의 관계

현재 `terraform/` 코드에는 **`count`를 쓰지 않는다** (`app_server` 1대, IAM attachment는 리소스 블록을 여러 개로 나눔).

강의에서 EC2를 N대로 늘리는 실습을 한다면 예:

```hcl
resource "aws_instance" "app_server" {
  count = 2

  ami                    = data.aws_ami.app.id
  instance_type          = "t2.micro"
  iam_instance_profile   = aws_iam_instance_profile.app_ec2_profile.name
  # ... 기존 인자 ...

  tags = {
    Name = "${var.project}-${var.environment}-app-${count.index}"
  }
}
```

SSM parameter·`depends_on` 등을 붙일 때는 **`aws_instance.app_server[count.index]`** 형태로 맞춘다.

---

## `count`의 한계 (75번으로 이어짐)

| 상황 | `count` | 비고 |
|------|---------|------|
| **개수만** 정해져 있고 내용이 같음 | 적합 | 슬라이드 EC2 4대 예 |
| **이름·설정이 항목마다 다름** | `count.index`로 분기해야 해서 지저분해짐 | **`for_each`** 가 보통 낫다 |
| **중간 항목만 삭제** | 인덱스가 밀리면 **다른 인스턴스가 replace** 될 수 있음 | `for_each`는 키 기준이라 안정적 |

같은 IAM policy를 4개 attachment로 붙이는 것처럼, **이미 리소스 블록을 여러 개** 쓴 경우는 `count` 없이도 복수 생성과 같은 효과다.

---

## `depends_on` · `lifecycle`과 함께

`count`는 **메타 인수**이므로 같은 블록에 `depends_on`, `lifecycle`을 둘 수 있다.

```hcl
resource "aws_instance" "server" {
  count = 2

  depends_on = [aws_iam_role_policy_attachment.app_iam_role_ssm_managed]

  lifecycle {
    create_before_destroy = true
  }
}
```

`depends_on` 대상이 `count` 있는 리소스면 **`resource.name[0]`** 처럼 인덱스를 붙이거나, attachment처럼 **인덱스 없는 단일 리소스**를 가리킨다.

---

## 정리

- **`count`**: 숫자만큼 **동일 블록**을 반복 생성.
- **`count.index`**: 0부터의 번호로 태그·이름 구분.
- 주소는 **`리소스타입.이름[인덱스]`**, 목록 참조는 **`[*]`**.
- **`for_each`와 동시 사용 불가** — 맵/이름 기준 복제는 75번.
- 다음: **75 `for_each`**.
