# 섹션 7 VPC·네트워크 — 이해한 것 / 헷갈렸던 것

> 파일 이름은 그대로 두었고, 맞춤법으로는 **「헷갈리는」** 이에요.

---

## 내가 이해한 강의 내용 (맞는 그림)

- **VPC** 안에 **퍼블릭 서브넷**과 **프라이빗 서브넷**이 있다.
- 서브넷은 **`aws_route_table_association`** 으로 **라우트 테이블**에 붙는다.
  - 퍼블릭 서브넷 → **퍼블릭용 라우트 테이블**
  - 프라이빗 서브넷 → **프라이빗용 라우트 테이블**
- **퍼블릭 라우트 테이블**에는 **`0.0.0.0/0` → 인터넷 게이트웨이(IGW)** 같은 라우트가 들어가서, 그 테이블을 쓰는 쪽이 **인터넷(외부)과 통신**할 수 있다.
- **프라이빗 라우트 테이블**에는 보통 IGW로 나가는 기본 라우트를 넣지 않는다. (VPC 내부 `local` 위주)
- Terraform에서는 **VPC / 서브넷 / 라우트 테이블 / 연결 / IGW / 라우트**가 **서로 다른 리소스**로 쪼개져 있다.

---

## 헷갈렸던 부분과 정리

### 1. “라우팅 테이블이 하나만 보인다” / Name 이 `-`

- **VPC를 만들 때 AWS가 자동으로 만드는 `메인(기본)` 라우트 테이블**이 항상 하나 있다. 이름 태그를 안 붙이면 콘솔 **Name** 칸이 **`-`** 로 나온다.
- 강의에서 Terraform으로 만드는 **`tastylog-dev-public-rt`**, **`…-private-rt`** 는 **그에 더해지는 테이블**이다. 그래서 같은 VPC에 **메인 1 + 커스텀 2**처럼 **3개**가 되는 게 정상에 가깝다.
- 목록에서 **다른 행**을 고르면 라우팅 탭 내용이 완전 달라 보인다. (메인은 `local`만, 퍼블릭 RT는 `0.0.0.0/0` → IGW 등)

### 2. “영상이랑 콘솔이 다르다”

- **리전·계정·VPC·라우트 테이블 ID**(`vpc-…`, `rtb-…`)는 영상과 **절대 같을 수 없다**. 다르다고 해서 잘못된 건 아니다.
- 영상은 보통 **이름 있는 퍼블릭 RT**를 연다. 내가 **`기본: 예` 인 메인 RT**만 보면 **IGW 라우트가 없어서** 영상과 다르게 느껴진다.
- **리소스 맵**에서 서브넷·RT에 마우스를 올리면 **실제 연결**이 강조되어 확인하기 좋다.

### 3. Terraform state / CLI 와 콘솔

- `terraform state list`, `terraform state show`, `aws ec2 describe-route-tables` 가 **API 기준 실제 연결**에 가깝다.
- 콘솔 UI는 필터·탭·지연 때문에 한눈에 안 맞아 보일 수 있다. **언짢으면 CLI로 한 번 맞춰 본 것**이 도움이 됐다.

### 4. 서브넷 CIDR 가 VPC 밖이면 에러 (`InvalidSubnet.Range`)

- 서브넷 `cidr_block` 은 **반드시 그 VPC의 `cidr_block` 안**이어야 한다. (예: VPC가 `10.0.0.0/20` 인데 서브넷만 `192.168.x.0/24` 이면 실패)

### 5. (참고) 메인 라우트 테이블은 꼭 “쓰는” 건 아님

- 모든 서브넷을 **커스텀 RT에 명시적으로 연결**해 두면, 메인 RT에는 **서브넷 연결이 없을 수도** 있다. 이상 현상이 아니라 **의도된 구성**일 수 있다.

---

## 앞으로 헷갈릴 때 짧은 체크리스트

1. **리전**이 강의·Terraform·콘솔이 같은지  
2. 라우팅 테이블 목록에서 **`기본: 예` vs `아니오`**, **Name** 으로 **어느 행**을 보는지  
3. **퍼블릭 RT** 라우팅 탭에 **`0.0.0.0/0` → `igw-…`** 가 있는지  
4. **`terraform plan` / `describe-route-tables`** 로 연결 관계 한 번 더 확인
