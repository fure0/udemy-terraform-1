# 70. AP 서버 구축

섹션14는 **AP(애플리케이션) 서버를 수동으로 구축한 뒤**, 그 상태를 **AMI로 남겨** 이후 EC2를 같은 이미지로 띄우는 흐름이다.  
이 메모는 **전체 구축 단계**와, **AMI로 만들기까지(전반부)** 범위를 정리한다.

---

## AP 서버 구축 4단계 (슬라이드 전체)

| 단계 | 내용 | 세부 |
|------|------|------|
| **1. OS 선택·기동** | 베이스 이미지로 EC2 준비 | ① AMI 선택 ② EC2 기동 |
| **2. OS·미들웨어 설정** | 실행 환경 | ① 미들웨어: **Node.js**, **jq** ② 서비스: **환경 변수 취득**(Parameter Store 등) |
| **3. 앱 설정** | 애플리케이션 배포 | ① 자재(소스·빌드물) 취득 ② 배포 ③ 서비스 기동 |
| **4. 공개** | 서비스 공개 | 슬라이드상 **C/O(Cutover)** 구간 이후 — 외부에 공개 가능한 상태 |

1~2는 **공통 OS·미들웨어·설정** 위주, 3~4는 **앱·릴리스** 위주로 나뉜다.

---

## AMI로 만드는 범위: 「전반부까지」

강의 **「前半まで構築してAMIにする」** 에 따르면, **AMI 이름 예: `tastylog-app-ami`** 로 남기는 시점은 **4단계 전체가 아니라 1~2단계까지**다.

```
[ AMI에 넣는 범위 ]
  1. OS 선택·기동 (AMI 선택 → EC2 기동)
  2. OS·미들웨어 설정
       - Node.js, jq 설치
       - 환경 변수 취득 (SSM Parameter Store — 섹션13)

[ AMI 이후 / 인스턴스마다 할 수 있는 작업 ]
  3. 앱 설정 (자재 취득·배포·서비스 기동)
  4. 공개 (C/O 이후)
```

- **AMI에 bake** 하는 것: OS + Node.js + jq + Parameter Store에서 DB 접속 정보 등을 **읽을 수 있는 준비**(스크립트·systemd·env 로딩 방식 등은 강의 실습 따름).
- **AMI에 넣지 않는 것(또는 나중)**: 특정 버전 앱 배포·최종 공개 — 인스턴스 기동 후 user data·배포 파이프라인 등으로 처리하는 그림이 일반적이다.

## 이전 섹션과의 연결

| 섹션 | AP 구축에서 쓰는 것 |
|------|---------------------|
| 10 | EC2, 키페어, 보안 그룹, 서브넷 |
| 12 | EC2 **IAM Role** → SSM·S3 등 |
| 13 | Parameter Store **`MYSQL_*`** → 환경 변수 취득 |

구축용 EC2는 Terraform으로 이미 띄운 **`app_server`** 를 쓰거나, 강의대로 **별도 수동 EC2**에서 작업한 뒤 AMI를 만든다.

---

## 실습 시 체크

- **Node.js**: 앱 런타임
- **jq**: JSON 처리(환경 변수·메타데이터 파싱 등)
- **환경 변수**: `/tastylog/dev/app/MYSQL_*` 등 SSM 경로에서 조회(IAM에 **SSM ReadOnly** 등 필요)
- AMI 생성 전: 불필요한 임시 파일·시크릿 로그 정리, **일관된 호스트명·태그** 확인

---

## 정리

- AP 서버 구축은 **OS → 미들웨어 → 앱 → 공개** 4단계.
- **`tastylog-app-ami`** 는 **1~2단계까지 반영한 골든 이미지**로 이해하면 된다.
- 다음 메모에서는 **AMI 생성·Terraform에서 custom AMI 참조** 등으로 이어진다.
