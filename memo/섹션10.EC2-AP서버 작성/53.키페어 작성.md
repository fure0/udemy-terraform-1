# 53. 키페어 작성

EC2에 SSH로 접속하려면 **키페어(공개키·비밀키)**가 필요하다. 강의에서는 크게 두 가지 흐름이 나온다.

---

## 키페어가 어떻게 나뉘는지

| 위치 | 비밀키(개인키) | 공개키 |
|------|----------------|--------|
| **콘솔에서 키페어 생성** | AWS가 만들 때 **한 번만 다운로드** → 로컬에 보관 | AWS(EC2 키페어)에 저장 |
| **로컬에서 `ssh-keygen` 후 Terraform 등록** | **로컬에만** 둔다(업로드하지 않음) | `aws_key_pair`로 **AWS에 등록** |

- **비밀키**는 본인 PC에만 두고, 유출되면 해당 키로 붙을 수 있으므로 권한·백업을 엄격히 한다.
- **공개키**만 AWS에 올리면 된다(두 번째 다이어그램 흐름).

---

## `ssh-keygen`으로 로컬에서 생성

인증용 키를 생성·관리·변환하는 도구다.

```bash
ssh-keygen [-t <TYPE>] [-b <BITS>] [-f <KEYFILE>]
```

| 인자 | 설명 |
|------|------|
| `-t <TYPE>` | 알고리즘 종류. 예: `rsa`, `ed25519` 등 |
| `-b <BITS>` | 비트 수(RSA 등에서 키 길이) |
| `-f <KEYFILE>` | 저장할 키 파일 경로·이름(확장자 없이 지정하면 `.pub`가 공개키) |

예시:

```bash
ssh-keygen -t rsa -b 4096 -f ~/.ssh/myapp-ec2 -N ""
chmod 400 ~/.ssh/myapp-ec2
```

공개키 내용은 `.pub` 파일을 열어 복사한다.

```bash
cat ~/.ssh/myapp-ec2.pub
```

---

## Terraform `aws_key_pair`

AWS 리전에 **공개키를 등록**해 EC2가 부팅 시 `~/.ssh/authorized_keys`에 쓸 수 있게 한다.

| 항목 | 타입 | 설명 |
|------|------|------|
| `key_name` | string | 콘솔에 보이는 **키페어 이름** |
| `public_key` | string | **공개키 문자열**(필수에 가깝게 사용) |
| `tags` | map(string) | 태그 |

```hcl
resource "aws_key_pair" "app" {
  key_name   = "${var.project}-${var.environment}-app"
  public_key = file("${path.module}/keys/myapp-ec2.pub")

  tags = {
    Name = "${var.project}-${var.environment}-app-key"
  }
}
```

`file()`로 `.pub`를 읽거나, `tls_private_key` 등으로 생성한 공개키를 넣는 방식도 있다. **비밀키는 Terraform 코드·state에 넣지 않는다.**

---

## 강의에서 말하는 작업 순서

1. **`ssh-keygen`으로 키페어 생성** (로컬에 비밀키·공개키)
2. **`aws_key_pair` 리소스를 코드에 작성** (`key_name`, `public_key`)
3. **`terraform apply`로 AWS에 키페어 등록**

이후 `aws_instance`의 `key_name`에 위 `aws_key_pair`의 이름을 지정하면 해당 공개키로 SSH 접속이 가능하다.

---

## 정리

- 콘솔 생성: 비밀키는 **다운로드 파일 하나**로만 받을 수 있고, 분실 시 같은 키는 복구 불가.
- Terraform + 로컬 생성: **공개키만** AWS에 올리고 비밀키는 로컬 관리가 일반적이다.
- 키 재생성·교체 시에는 EC2에 새 키를 반영하는 절차(AMI·User Data·수동 교체 등)가 따로 필요하다.
