# 75. 복수 리소스 생성 (`for_each`)

**map 또는 set**을 펼치면서, 항목마다 리소스를 하나씩 만든다.  
74번 `count`가 **개수(인덱스)** 기준이라면, `for_each`는 **키(이름)·값 쌍** 기준이라 **항목별 설정이 다를 때** 유리하다.

이전: **74 `count`** · 다음: **76 `lifecycle`**

---

## 복수 리소스를 만드는 방법 (슬라이드)

| 종류 | 설명 |
|------|------|
| **`count`** | **지정한 횟수**만큼 복제 |
| **`for_each`** | **리스트·맵(set)** 을 돌며 복제 |

**`count`와 `for_each`는 동시에 지정할 수 없다.**

---

## `for_each`란

**지정한 map 또는 set of strings**를 펼치며 복수 리소스를 생성한다.

슬라이드: 「指定された map または set を展開しながら複数リソース生成する」

### map 예 — 서브넷 (슬라이드)

키 = CIDR, 값 = AZ 로 **대마다 다른 값**을 넣기 좋다.

```hcl
resource "aws_vpc" "vpc" {
  cidr_block = "192.168.0.0/20"
}

resource "aws_subnet" "subnet" {
  for_each = {
    "192.168.1.0/24" = "ap-northeast-1a"
    "192.168.2.0/24" = "ap-northeast-1c"
    "192.168.3.0/24" = "ap-northeast-1d"
  }

  vpc_id            = aws_vpc.vpc.id
  cidr_block        = each.key
  availability_zone = each.value
}
```

| 심볼 | map일 때 | set일 때 |
|------|----------|----------|
| **`each.key`** | map의 **키** (여기서 CIDR) | set 원소 (문자열) |
| **`each.value`** | map의 **값** (여기서 AZ) | `each.key`와 동일 |

생성 결과: 서브넷 **3개**, 주소는 예:

```hcl
aws_subnet.subnet["192.168.1.0/24"]
aws_subnet.subnet["192.168.2.0/24"]
aws_subnet.subnet["192.168.3.0/24"]
```

`count`의 `[0]`, `[1]` 대신 **문자열 키**로 가리킨다.

---

## 배열(리스트)을 쓰고 싶을 때 — `toset`

`for_each`는 **list를 직접 받지 않는다.**  
문자열 리스트는 **`toset()`** 으로 set으로 바꾼 뒤 쓴다.

슬라이드: 「配列を指定したい場合 toset 関数を利用して set へ変換する」

```hcl
resource "aws_iam_user" "user" {
  for_each = toset([
    "tanaka",
    "sato",
  ])

  name = each.key
}
```

- `each.key` / `each.value` 모두 `"tanaka"`, `"sato"` (set은 값이 키와 같음)
- 주소: `aws_iam_user.user["tanaka"]`, `aws_iam_user.user["sato"]`

**주의:** `toset` 변환 시 **중복 문자열은 제거**된다. 순서는 보장되지 않는다(실행 순서에 의존하지 말 것).

---

## `count` vs `for_each` (74번 보완)

| | `count` | `for_each` |
|---|---------|------------|
| **기준** | 숫자 `N`, `count.index` | map/set의 **키** |
| **주소** | `resource.name[0]` | `resource.name["키"]` |
| **중간 삭제** | 인덱스가 밀리면 **다른 인스턴스가 replace** 될 수 있음 | **키가 같으면** 같은 리소스로 유지 |
| **적합** | 개수만 다르고 설정이 거의 같음 | CIDR·AZ·policy ARN처럼 **항목마다 값이 다름** |

---

## 여러 개 참조하기

```hcl
# map/set for_each 전체의 id 목록
output "subnet_ids" {
  value = [for k, s in aws_subnet.subnet : s.id]
}

# 또는 values() / keys() 조합
# values(aws_subnet.subnet)[*].id  등
```

다른 리소스에서 특정 하나만:

```hcl
aws_subnet.subnet["192.168.1.0/24"].id
```

## 이 프로젝트에 적용하는 예 (`iam.tf`)

지금은 **policy attachment 블록을 4개** 나눠 썼다. `for_each`로 묶을 수 있다.

```hcl
locals {
  app_role_managed_policies = toset([
    "arn:aws:iam::aws:policy/AmazonEC2ReadOnlyAccess",
    "arn:aws:iam::aws:policy/AmazonSSMManagedInstanceCore",
    "arn:aws:iam::aws:policy/AmazonSSMReadOnlyAccess",
    "arn:aws:iam::aws:policy/AmazonS3ReadOnlyAccess",
  ])
}

resource "aws_iam_role_policy_attachment" "app_iam_role" {
  for_each = local.app_role_managed_policies

  role       = aws_iam_role.app_iam_role.name
  policy_arn = each.key
}
```

- 주소 예: `aws_iam_role_policy_attachment.app_iam_role["arn:aws:iam::aws:policy/AmazonSSMManagedInstanceCore"]`
- EC2 `depends_on`을 쓸 때는 **한 attachment**가 아니라 `depends_on`에 **리소스 타입 전체** 또는 필요한 키를 지정하는 방식으로 코드를 맞춰야 한다(리팩터 시 주의).

강의 실습 전까지는 **기존 4블록** 그대로 두어도 된다.

---

## 제약·팁

- `for_each`의 키는 **plan 시점에 정해진 문자열**이어야 한다. 계산 결과가 바뀌면 리소스 주소가 바뀌고 **destroy/create**가 날 수 있다.
- **빈 map/set** → 리소스 0개 (`count = 0`과 비슷).
- **모듈**에도 `for_each` 사용 가능 (`module "x" { for_each = ... }`).
- map 키에 **민감한 값만** 넣지 말 것 — state·주소에 키가 남는다.

---

## 정리

- **`for_each`**: map 또는 **set of strings**로 복수 생성.
- 리스트는 **`toset([...])`** 로 변환.
- 블록 안에서는 **`each.key`**, **`each.value`** (map).
- 참조는 **`리소스.이름["키"]`** — 인덱스보다 **이름·ARN 기준**으로 안정적.
- 다음: **76 `lifecycle`**.
