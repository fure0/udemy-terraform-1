# 73. 리소스 의존 정의 (`depends_on`)

**Terraform 리소스 의존 관계 제어** — 어떤 리소스를 **먼저** 만들고 **나중에** 만들지 정하는 주제다.  
대부분은 **속성 참조**만으로 자동 판별되고, **`depends_on`** 은 그래프에 안 잡히는 관계를 **수동으로** 넣을 때 쓴다.

이전: **72 메타 인수** · 다음: **74 `count`**

---

## 참조와 의존 (슬라이드 다이어그램)

화살표는 **참조 원(参照元) → 참조 대상(参照先)** 이다.  
Terraform은 **참조 대상이 먼저** 준비된 뒤 **참조 원**을 처리한다.

| 아이콘 | 의미 |
|--------|------|
| 다이아몬드 | `resource` |
| 문서 | `data` |

### IAM Role 예 (섹션12와 동일 구조)

```
인스턴스 프로필 ──→ IAM Role ──→ 신뢰 정책 (data)
                      ↑
        Role Policy Attachment ──→ 정책 (data/ARN)
```

| 참조 원 | 참조 대상 | 코드에서의 연결 |
|---------|-----------|-----------------|
| `aws_iam_instance_profile` | `aws_iam_role` | `role = aws_iam_role.app_iam_role.name` |
| `aws_iam_role` | `data.aws_iam_policy_document` | `assume_role_policy = data....json` |
| `aws_iam_role_policy_attachment` | `aws_iam_role` | `role = aws_iam_role....name` |

이 저장소 예:

```hcl
resource "aws_iam_instance_profile" "app_ec2_profile" {
  name = aws_iam_role.app_iam_role.name
  role = aws_iam_role.app_iam_role.name
}
```

`role = aws_iam_role.app_iam_role.name` 처럼 **다른 리소스·data 속성을 읽으면** Terraform이 **암시적 의존(implicit dependency)** 을 그린다.  
슬라이드: **「기본적으로는 자동으로 판별된다」**.

---

## `depends_on` 이란

**Terraform이 코드만 보고는 감지하지 못하는 의존 관계**를 `resource` 블록에 명시한다.

슬라이드 표현: AWS 쪽에는 **상·하위 관계**가 있는데, HCL에는 **서로를 참조하지 않아** 그래프에 선이 없는 경우.

### 문법

```hcl
resource "<RESOURCE_TYPE>" "<RESOURCE_NAME>" {
  ...

  depends_on = [
    <ADDRESS>
  ]
}
```

- `<ADDRESS>`: 리소스 주소. 예) `aws_iam_role.app_iam_role`, `aws_s3_bucket_public_access_block.example`
- **리스트**로 여러 대상을 넣을 수 있다.
- `depends_on` 안의 리소스가 **완료된 뒤** 이 리소스의 create/update가 진행된다.

---

## S3 예 (슬라이드)

```
Bucket ← Public Access Block
  ↑
Bucket Policy ← Policy Document (data)
```

| 리소스 | 참조 대상 | 자동 의존 |
|--------|-----------|-----------|
| 버킷 | (없음, 루트) | — |
| Public Access Block | 버킷 ID/이름 | O |
| Bucket Policy | 버킷 + policy JSON | O (버킷·data 쪽) |

버킷·policy document를 **속성으로 참조**하면 Bucket / data는 자동으로 먼저 처리된다.

### `depends_on` 이 필요한 경우 (슬라이드 강조)

**Public Access Block** 과 **Bucket Policy** 는 둘 다 같은 버킷을 가리키지만, **서로의 리소스를 참조하지 않을 수 있다**.

- **Terraform 리소스 그래프**: 둘 사이에 **직접 의존 없음** → 동시에 apply 시도 가능
- **AWS 개념**: 설정 순서·상태에 따라 **한쪽이 먼저** 맞춰져야 policy 적용이 안정적인 경우가 있음

이때 Bucket Policy(또는 먼저 맞춰야 하는 쪽)에 예:

```hcl
resource "aws_s3_bucket_policy" "example" {
  bucket = aws_s3_bucket.example.id
  policy = data.aws_iam_policy_document.bucket.json

  depends_on = [
    aws_s3_bucket_public_access_block.example
  ]
}
```

**코드에 참조가 없어도** 생성 순서를 강제할 때 쓴다.

## 암시적 vs `depends_on` (정리)

| 구분 | 감지 방법 | 예 |
|------|-----------|-----|
| **암시적 의존** | 다른 리소스 **속성**을 인자에 사용 | `role = aws_iam_role.app_iam_role.name` |
| **명시적 의존** | `depends_on` 목록 | S3 policy ↔ public access block |

**원칙**: 참조로 잡히면 **`depends_on` 없이** 두는 것이 좋다.  
같은 대상을 `depends_on`과 속성 참조 **둘 다**로 중복 지정할 필요는 없다.

## 이 프로젝트에서 자주 나오는 패턴

### 1. IAM — 대부분 자동

`iam.tf`의 프로필·Role·attachment는 `role =`, `assume_role_policy =` 로 이미 연결되어 있어 **추가 `depends_on` 없이** Role → Attachment → Profile 순이 잡힌다.

### 2. EC2 + Role 권한 — 참조만으로는 부족할 때

EC2는 **인스턴스 프로필 이름**만 넣고, **어떤 managed policy가 붙었는지**는 참조하지 않는다.  
Role에 policy attachment가 **끝나기 전**에 인스턴스가 올라가면, 부팅 직후 API 호출이 실패할 수 있어 강의·72번 예시처럼:

```hcl
resource "aws_instance" "app_server" {
  ...
  iam_instance_profile = aws_iam_instance_profile.app_ec2_profile.name

  depends_on = [
    aws_iam_role_policy_attachment.app_iam_role_ssm_managed,
    # 필요 시 다른 attachment도 나열
  ]
}
```

**프로필 → Role** 은 `iam_instance_profile` 참조로 자동이지만, **attachment 완료**는 그래프에 안 보일 수 있어 `depends_on`으로 보완한다.

### 3. SSM Parameter — RDS 주소 등

`appserver.tf`의 `aws_ssm_parameter`는 `value = aws_db_instance.mysql_standalone.address` 등으로 **RDS에 암시적 의존**이 있다. RDS가 없으면 parameter도 만들 수 없다.

---

## 주의

- `depends_on`은 **생성·삭제 순서**만 바꾼다. **런타임 동작**(앱이 DB에 붙는 타이밍 등)까지 보장하지는 않는다.
- 과도하게 쓰면 그래프가 복잡해지고 apply가 **직렬**로 길어질 수 있다.
- **모듈 전체**에 의존하려면 `module.xxx` 주소를 쓸 수 있다(모듈 단위 완료 후 진행).

---

## 정리

- **참조(속성 사용)** 가 있으면 → **자동 의존** (IAM 프로필 → Role 등).
- **참조 없이 순서만 필요**하면 → **`depends_on`** (S3 policy vs public access block, EC2 vs policy attachment 등).
