# 40. 프리픽스 취득

## 1) 프리픽스 리스트(Prefix List)란?

- 여러 CIDR 블록을 하나의 묶음으로 관리하는 개념
- AWS 관리형 서비스(S3 등)의 IP 대역은 변경될 수 있으므로, 고정 CIDR 직접 입력보다 Prefix List 활용이 안전하다.

예시 흐름:

- EC2에서 S3에 접근할 때
- 실제로는 S3 도메인 뒤에 여러 IP 대역(CIDR)이 존재
- 이 CIDR 묶음을 Prefix List로 참조하면 SG/NACL 정책 관리가 쉬워진다.

## 2) Terraform에서 Prefix List 조회

프리픽스 리스트 정보는 데이터 소스 `aws_prefix_list`로 취득한다.

기본 형태:

```hcl
data "aws_prefix_list" "<DATA_NAME>" {
  # prefix_list_id 또는 name으로 조회
}
```

주요 조회 키:

- `prefix_list_id`: 프리픽스 리스트 ID로 조회
- `name`: 프리픽스 리스트 이름으로 조회

## 3) 현재 코드 적용 예시

`terraform/data.tf`

```hcl
variable "region" {
  type    = string
  default = "ap-northeast-1"
}

data "aws_prefix_list" "s3_pl" {
  name = "com.amazonaws.${var.region}.s3"
}
```

의미:

- 현재 리전에 맞는 S3 관리형 Prefix List를 1건으로 조회
- 이후 `data.aws_prefix_list.s3_pl.id` 형태로 SG 규칙 등에 연결 가능

## 4) 이번 에러 원인 정리

에러 메시지:

- `multiple EC2 Prefix Lists matched`

원인:

- `name = "com.amazonaws.*.s3"`처럼 와일드카드 형태로 쓰면
- 복수 Prefix List가 동시에 매칭되어 Terraform이 단일 결과를 고르지 못함

해결:

- 리전을 명시해 단일 항목으로 좁힘  
  (`com.amazonaws.ap-northeast-1.s3` 또는 `com.amazonaws.${var.region}.s3`)

## 5) 실무 팁

- 강의와 다르게 에러가 나는 경우, 대부분 리전/계정/Provider 버전 차이로 조회 결과가 달라진다.
- 데이터 소스는 "정확히 1건" 조회되도록 조건을 구체화하는 습관이 중요하다.
