# 39. SG그룹 작성1

`terraform/security_group.tf` 기준으로 Security Group을 역할별로 분리해 작성한다.

## 1) 생성된 SG 리소스

- `aws_security_group.web_sg`: 웹(프론트) 계층용 SG
- `aws_security_group.app_sg`: 애플리케이션 계층용 SG
- `aws_security_group.opmng_sg`: 운영/관리(배스천, 운영 접속) 계층용 SG
- `aws_security_group.db_sg`: 데이터베이스 계층용 SG

공통적으로 아래 값을 사용한다.

- `name`: `${var.project}-${var.environment}-<role>-sg`
- `vpc_id`: `aws_vpc.vpc.id`
- `tags`: `Name`, `Project`, `Env`

## 2) web_sg 규칙

### Ingress
- `80/tcp` 허용 (`0.0.0.0/0`)
- `443/tcp` 허용 (`0.0.0.0/0`)

### Egress
- `3000/tcp`를 `app_sg`로만 허용  
  (`source_security_group_id = aws_security_group.app_sg.id`)

즉, 인터넷에서 웹 서버로는 80/443 접근을 받고, 웹 계층에서 앱 계층으로는 3000 포트로만 통신하도록 제한한다.

## 3) opmng_sg 규칙

### Ingress
- `22/tcp` 허용 (`0.0.0.0/0`)
- `3000/tcp` 허용 (`0.0.0.0/0`)

### Egress
- `80/tcp` 허용 (`0.0.0.0/0`)
- `443/tcp` 허용 (`0.0.0.0/0`)

운영/관리 서버에서 외부 패키지 저장소 접근(80/443)과 SSH 접속(22) 용도를 의도한 구성이다.

## 4) db_sg 규칙

### Ingress
- `3306/tcp`를 `app_sg`에서만 허용  
  (`source_security_group_id = aws_security_group.app_sg.id`)

DB 포트는 인터넷 전체가 아니라 앱 서버 SG를 소스로 제한해 최소 권한 원칙을 적용한다.

## 5) 흐름 요약

- 사용자 → `web_sg` : 80/443
- `web_sg` → `app_sg` : 3000
- `app_sg` → `db_sg` : 3306

계층 간 통신을 `cidr_blocks` 대신 `source_security_group_id`로 연결해, IP 변경에도 정책을 안정적으로 유지할 수 있다.

## 6) 보안 관점 체크 포인트

- 운영 환경에서는 `0.0.0.0/0`로 열린 `22`, `3000`은 관리망 IP 대역으로 좁히는 것이 안전하다.
- SG 이름/태그에 환경(`dev`, `stg`, `prod`)을 포함해 운영 시 추적성을 높인다.
