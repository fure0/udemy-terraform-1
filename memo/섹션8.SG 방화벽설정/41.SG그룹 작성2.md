# 41. SG그룹 작성2

이번 단계에서는 `app_sg` 중심으로 애플리케이션 계층의 통신 경로를 구체화한다.

## 1) app_sg 인바운드 규칙 추가

```hcl
resource "aws_security_group_rule" "app_in_tcp3000" {
  security_group_id        = aws_security_group.app_sg.id
  type                     = "ingress"
  protocol                 = "tcp"
  from_port                = 3000
  to_port                  = 3000
  source_security_group_id = aws_security_group.web_sg.id
}
```

- 앱 서버는 `3000/tcp`를 `web_sg`에서만 허용
- `cidr_blocks` 대신 `source_security_group_id`를 써서, 웹 계층에서 오는 트래픽만 정확히 허용

## 2) app_sg 아웃바운드 규칙(S3 접근) 추가

```hcl
resource "aws_security_group_rule" "app_out_http" {
  security_group_id = aws_security_group.app_sg.id
  type              = "egress"
  protocol          = "tcp"
  from_port         = 80
  to_port           = 80
  prefix_list_ids   = [data.aws_prefix_list.s3_pl.id]
}

resource "aws_security_group_rule" "app_out_https" {
  security_group_id = aws_security_group.app_sg.id
  type              = "egress"
  protocol          = "tcp"
  from_port         = 443
  to_port           = 443
  prefix_list_ids   = [data.aws_prefix_list.s3_pl.id]
}
```

- 앱 서버의 외부 통신을 무조건 `0.0.0.0/0`로 열지 않고, S3 Prefix List로 제한
- 즉, HTTP/HTTPS라도 목적지를 S3 관련 주소 대역으로 좁힌다.

## 3) app_sg 아웃바운드 규칙(DB 접근) 추가

```hcl
resource "aws_security_group_rule" "app_out_tcp3306" {
  security_group_id        = aws_security_group.app_sg.id
  type                     = "egress"
  protocol                 = "tcp"
  from_port                = 3306
  to_port                  = 3306
  source_security_group_id = aws_security_group.db_sg.id
}
```

- 앱 서버에서 DB로 `3306/tcp` 연결 허용
- 목적지를 DB SG로 제한해 계층 간 통신을 명확히 분리

## 4) 전체 통신 구조(작성2 시점)

- 사용자 → `web_sg` : `80/443`
- `web_sg` → `app_sg` : `3000`
- `app_sg` → `db_sg` : `3306`
- `app_sg` → `S3 Prefix List` : `80/443`

## 5) 정리 포인트

- 계층 간 내부 통신은 SG 참조(`source_security_group_id`)로 제어
- AWS 관리형 서비스(S3) 통신은 `prefix_list_ids`로 제어
- 결과적으로 "열어야 할 경로만 여는" 최소 권한 보안 구성이 된다.
