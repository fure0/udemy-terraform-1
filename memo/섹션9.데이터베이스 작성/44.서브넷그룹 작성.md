# 44. 서브넷그룹 작성

## 1) RDS에서 서브넷그룹이 필요한 이유

RDS 인스턴스는 `VPC` 안의 특정 서브넷에 배치된다.  
하지만 “어떤 서브넷들을 묶어서 DB를 배치할지”를 한 번에 지정해야 하며, 그 역할을 하는 것이 **서브넷 그룹(DB Subnet Group)** 이다.

강의 흐름 기준으로 RDS 구성 요소는 다음 3가지로 정리된다.

- 파라미터 그룹: DB 엔진 동작값(예: 문자셋)
- 옵션 그룹: 엔진에 추가 기능(예: 플러그인/옵션)
- 서브넷 그룹: DB를 배치할 서브넷 묶음(네트워크 범위)

## 2) `aws_db_subnet_group` 주요 속성

Terraform 리소스: `aws_db_subnet_group`

필드(강의 이미지 기준):

- `name`: 서브넷 그룹 이름
- `subnet_ids`: DB를 배치할 서브넷 ID 배열
- `tags`: 태그(식별용 메타데이터)

## 3) 현재 코드 작성 내용

`terraform/rds.tf`

```hcl
resource "aws_db_subnet_group" "mysql_standalone_subnetgroup" {
  name = "${var.project}-${var.environment}-mysql-standalone-subnetgroup"
  subnet_ids = [
    aws_subnet.private_subnet_1a.id,
    aws_subnet.private_subnet_1c.id
  ]

  tags = {
    Name    = "${var.project}-${var.environment}-mysql-standalone-subnetgroup"
    Project = var.project
    Env     = var.environment
  }
}
```

## 4) 설정 의도

- `aws_subnet.private_subnet_1a`, `aws_subnet.private_subnet_1c`처럼 **private subnet** 2개를 묶는다.
- 이렇게 하면 RDS가 멀티 AZ(가용영역) 구성에서 배치될 여지를 가지며, 외부에서 직접 접근 불가능한 네트워크 영역에 DB를 두는 방향이 된다.

## 5) 체크 포인트

- `subnet_ids`에 넣는 서브넷이 실제로 `private`인지(라우팅/인터넷 게이트웨이 여부) 확인
- 서브넷 그룹에 최소 2개 이상의 서브넷을 넣는 패턴이 일반적으로 안정적이다(특히 Multi-AZ 목적일 때)
