# 48. RDS 패스워드

Terraform으로 RDS를 만들 때 `password`를 리소스에 넣으면 **비밀번호가 `terraform.tfstate`에 평문(클리어 텍스트)으로 기록**된다는 점이 보안상의 큰 걱정거리다. 로컬에서 `apply`한 결과가 상태 파일에 그대로 남는다.

참고: 변수에 `sensitive = true`를 붙이면 CLI 출력에서는 숨겨지지만, **상태 파일에는 여전히 저장**된다. 상태는 원격 백엔드(S3 등)로 옮기고 암호화·IAM으로 접근을 제한하는 것도 기본 전제에 포함한다.

---

## 대응 방법 개요

| 종류 | 설명 |
|------|------|
| **나중에 패스워드 변경** | RDS 마스터 비밀번호를 콘솔·CLI 등으로 배포 **이후**에 직접 바꾼다. `tfstate`가 유출되어도 그 안의 비밀번호는 **이미 무효**일 수 있다. |
| **운용 회피(액세스 권한 제어)** | 관리자와 개발자 등 **역할별로** `tfstate`·비밀번호·파라미터를 볼 수 있는 권한을 나눈다. 불필요한 사람이 상태·시크릿을 읽지 못하게 한다. |

---

## 장단점 비교

| 종류 | 장점 | 단점 |
|------|------|------|
| **나중에 패스워드 변경** | 실제 운용 비밀번호가 `tfstate`와 일치하지 않게 만들 수 있어, 유출 시 위험을 줄일 수 있다. | `tfstate`와 AWS 실제 값이 **어긋나(drift)** 이후 `apply` 시 비밀번호를 다시 맞추려 하거나 관리가 꼬일 수 있다. **비밀번호는 별도 안전한 저장소**에서 관리해야 한다. |
| **운용 회피(액세스 권한 제어)** | Terraform만으로 흐름을 유지하기 쉽고, 코드·상태 기준으로 일관되게 관리할 수 있다. | **누가 상태·시크릿을 볼 수 있는지** 사람·IAM 단위로 엄격히 관리해야 한다. |

---

## 대응 ① 나중에 패스워드 변경

배포 후 관리자가 AWS CLI로 마스터 비밀번호만 바꾼다.

```bash
aws rds modify-db-instance \
  --db-instance-identifier '<RDS_INSTANCE_ID>' \
  --master-user-password '<NEW_PASSWORD>'
```

Terraform이 다음 `apply`에서 **코드에 있는(또는 상태에 남은) 비밀번호로 되돌리려 하지 않게** 하려면, `aws_db_instance`에 **`lifecycle` 블록**을 넣어 비밀번호 필드 변경을 무시한다(강의에서는 별도 섹션에서 상세).

```hcl
lifecycle {
  ignore_changes = [password]
}
```

실제 필드명은 엔진·리소스에 따라 `password` 외 표기가 있을 수 있으니, 해당 강의·문서의 예시를 그대로 따른다.

---

## 대응 ② 운용 회피(액세스 권한 제어)

슬라이드 예: 비밀번호 등을 **Systems Manager 파라미터 스토어**에 두고, **개발자**에게는 SSM 읽기 API를 막는 IAM 정책을 붙인다.

```json
"Effect": "Deny",
"Action": [
  "ssm:GetParameterHistory",
  "ssm:DescribeDocumentParameters",
  "ssm:GetParametersByPath",
  "ssm:GetParameters",
  "ssm:GetParameter"
]
```

**관리자**는 Terraform 워크스테이션 등에서만 파라미터·인프라를 다루게 하고, 개발자는 콘솔·CLI로 해당 시크릿을 읽지 못하게 하는 식의 **역할 분리**가 요지다. 동일한 원리로 **원격 상태 버킷·DynamoDB 잠금 테이블**에도 최소 권한을 적용한다.

---

## 정리

- Terraform으로 RDS `password`를 넣으면 **상태에 평문 저장**된다는 전제를 두고 설계한다.
- **①**은 배포 후 비밀번호 회전 + `ignore_changes` 등으로 Terraform과 실제 비밀번호의 관계를 정한다.
- **②**는 비밀번호를 코드·상태에 두더라도 **IAM·백엔드 권한**으로 노출 경로를 줄인다.

실무에서는 **Secrets Manager** 연동, RDS가 비밀을 직접 참조하는 패턴 등도 많이 쓰이며, 강의 슬라이드의 두 축(회전·권한)과 함께 이해하면 된다.
