# 71. AP 서버 구축 2

섹션15는 **「手動でAPサーバー構築を行います」** — AP 서버를 **수동으로** 구축하는 실습이다.  
섹션14(70번)에서 **AMI(`tastylog-app-ami`)** 로 **전반부(1~2단계)** 를 굳혀 두었다면, 여기서는 **후반부: 앱 배치·기동**에 해당한다(슬라이드: **後半のアプリ配置/起動**).

---

## 4단계 중 이번 범위

| 단계 | 섹션14 AMI | 섹션15 (이번) |
|------|------------|----------------|
| 1. OS 선택·기동 | AMI에 포함 | custom AMI로 EC2 기동(또는 기존 인스턴스) |
| 2. OS·미들웨어 | AMI에 포함 (Node.js, jq, 환경 변수) | 이미 갖춰진 상태에서 진행 |
| **3. 앱 설정** | AMI **밖** | **여기서 수행** |
| **4. 공개** | AMI **밖** | **C/O 이후** — 서비스 공개 |

---

## 3. 앱 설정 (슬라이드)

| 순서 | 작업 | 설명 |
|------|------|------|
| ① | **자재 취득** | 앱 소스·빌드 산출물(아카이브 등)을 EC2로 가져옴 (SCP, S3, git 등 — 강의 방식 따름) |
| ② | **자재 배포** | 압축 해제·디렉터리 배치·권한 설정 |
| ③ | **서비스 기동** | Node 앱을 **systemd** 등으로 기동, 로그·포트(예: 3000) 확인 |

전제: AMI(또는 1~2단계)에서 **Node.js·jq**와 **Parameter Store 기반 DB 환경 변수**가 이미 준비되어 있어야 앱이 RDS에 연결할 수 있다(섹션13·12).

---

## 4. 공개 (슬라이드)

슬라이드의 **세로 점선 C/O(Cutover)** 는 **구축·배포**와 **대외 공개**의 경계에 가깝다.

- 앱이 **의도한 포트**에서 응답하는지 확인
- **보안 그룹·ALB**(이후 섹션)를 통해 사용자·웹 계층에서 접근 가능한 상태로 넘김
- 「공개」는 인프라 관점에서 **트래픽이 AP까지 도달**하는 구성이 갖춰진 단계로 이해하면 된다

---

## 70번(섹션14)과의 관계

```
섹션14:  [1 OS] [2 미들웨어] ──→ AMI (tastylog-app-ami)
섹션15:  AMI로 EC2 기동 ──→ [3 앱] [4 공개]  (수동 구축)
```

- **같은 4단계 다이어그램**이지만, **70 = AMI까지**, **71 = 앱·공개까지 수동 완료**.
- Terraform `aws_instance`는 이후 **custom AMI**·user data 등으로 이 과정을 자동화하는 방향으로 이어질 수 있다.

---

## 실습 체크

- SSM **`/tastylog/dev/app/MYSQL_*`** 가 EC2에서 읽히는지
- 앱 프로세스 기동 후 **로컬 curl**·DB 연결 로그
- SG에서 **웹 → AP(3000)** 등 필요한 포트가 열려 있는지(이미 `app_sg` 등에 정의된 경우 확인)

---

## 정리

- **71 = AP 서버 구축의 후반**: 자재 취득 → 배포 → 서비스 기동 → (C/O 이후) 공개.
- **전반(미들웨어·env)** 은 AMI·섹션14, **후반(앱)** 은 섹션15 수동 실습.
- 다음에는 **Terraform으로 AP용 EC2·AMI 참조** 등 인프라 코드화로 넘어간다.
