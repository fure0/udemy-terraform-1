# 59. 리소스명 변경

`.tf`에서만 `resource "aws_instance" "web"`을 `"app"`으로 바꾸면, Terraform은 **같은 EC2를 계속 쓰는 것**이 아니라 **옛 주소 리소스 삭제 + 새 주소 리소스 생성**으로 해석할 수 있다.  
이유는 **상태(`tfstate`)의 리소스 주소**가 여전히 `aws_instance.web`이기 때문이다(슬라이드: 코드만 바꿔도 tfstate와 차이가 난다 → **tfstate도 맞춰야 한다**).

실제 인프라를 지우지 않고 **이름(상태상 주소)만 바꾸려면** `terraform state mv`로 상태를 옮긴다.

---

## 명령: 상태에서 주소 이동

```bash
terraform state mv <SOURCE> <DESTINATION>
```

| 인자 | 설명 |
|------|------|
| `<SOURCE>` | 이동 **전** 상태에 있는 리소스 주소 |
| `<DESTINATION>` | 이동 **후**에 쓰일 리소스 주소(코드의 새 `resource` 이름과 일치) |

AWS 리소스 자체를 복사하는 명령이 아니라, **tfstate 안의 해당 항목의 키(주소)만 바꾼다.** 실체는 그대로다.

---

## 예시 (EC2 리소스 블록 이름 변경)

```bash
terraform state mv \
  aws_instance.<SRC_NAME> \
  aws_instance.<DEST_NAME>
```

`<SRC_NAME>` / `<DEST_NAME>`은 각각 `resource "aws_instance" "..."`의 로컬 이름이다.

---

## 강의에서의 작업 순서

1. **코드에서** EC2 등 리소스 블록 이름을 `<SRC_NAME>` → `<DEST_NAME>`으로 바꾼 뒤, **`terraform state mv`**로 상태 주소를 맞춘다(위 예시).  
   그다음 `terraform plan`으로 **삭제·재생성이 아닌지** 확인한다.
2. **`terraform plan`**  
   `state mv`와 코드가 맞으면 보통 **변경 없음**에 가깝게 나온다(다른 수정이 없다면).
3. **원래 이름으로 되돌리기**  
   실습이면 다시 코드·`state mv`로 역순 복구해 본다.

> 순서 팁: 코드를 먼저 바꾼 직후에는 `plan`이 destroy/create로 보일 수 있다. 그때 **`apply` 하기 전에** `state mv`로 맞추면 안전하다.

---

## 정리

- 리소스 **로컬 이름 변경**은 **코드 + tfstate**를 같이 맞추는 작업이다.
- **`terraform state mv`**는 상태 주소만 바꾸므로, 리네임·리팩터링 시 자주 쓴다.
- 모듈 안 주소는 `module.foo.aws_instance.bar`처럼 길어질 수 있어, `terraform state list`로 정확한 주소를 복사하는 것이 좋다.
