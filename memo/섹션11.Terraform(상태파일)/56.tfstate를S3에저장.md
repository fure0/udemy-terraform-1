# 56. tfstate를 S3에 저장

로컬에만 `terraform.tfstate`를 두면, **사람마다 다른 상태 파일**을 갖게 되어 AWS 실제 리소스와 **tfstate 정합성이 깨지기** 쉽다. 팀이 같은 인프라를 다룰 때는 **관리 대상 시스템당 공유되는 상태 파일 하나**를 두는 것이 전제다.

---

## 왜 원격 상태(S3)인가

- **한 소스 오브 트루스**: 모든 구성원이 같은 `tfstate`를 읽고 쓴다.
- **백업·암호화·접근 제어**: S3 버킷 정책·버전 관리·SSE 등으로 보호하기 쉽다.
- (실무 확장) **DynamoDB 잠금 테이블**을 같이 쓰면 동시 `apply` 충돌을 줄일 수 있다. 강의 슬라이드는 S3 중심이면 이후 단계에서 다룰 수 있다.

슬라이드 권장: **Terraform이 만드는 리소스가 있는 계정**과 **상태를 둘 S3가 있는 계정**을 **분리**하면, 관리 밖 버킷이 본전처럼 취급되거나 잘못 적용되는 위험을 줄일 수 있다.  
강의에서는 학습 편의상 **같은 계정**에 버킷을 두는 경우가 많다.

---

## 관리 대상당 tfstate 하나

- 프로젝트(스택) 단위로 **버킷 내 `key` 경로**를 나누어 **상태 파일을 하나**로 공유한다.
- 여러 환경이면 `dev/terraform.tfstate`, `prod/terraform.tfstate`처럼 `key`로 분리하는 패턴이 흔하다.

---

## `terraform` 블록과 백엔드

Terraform의 **기본 설정**(버전 제약, 백엔드 등)은 루트 모듈의 **`terraform { ... }`** 안에 둔다. 여기에 **`backend "s3" { ... }`**를 넣어 상태 저장 위치를 지정한다.

| 항목 | 타입 | 설명 |
|------|------|------|
| `backend` | 블록 | 백엔드 종류 지정. `"s3"`일 때 아래 인자 사용 |
| `bucket` | string | 상태를 둘 S3 버킷 이름 |
| `key` | string | 버킷 안 객체 키(파일 경로). 예: `tastylog/dev/terraform.tfstate` |
| `region` | string | 버킷이 있는 리전 |
| `profile` | string | (선택) 사용할 AWS CLI 프로파일 |

```hcl
terraform {
  required_version = ">= 1.0.0"

  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = ">= 4.0.0"
    }
  }

  backend "s3" {
    bucket  = "my-terraform-state-bucket"
    key     = "tastylog/dev/terraform.tfstate"
    region  = "ap-northeast-1"
    profile = "terraform"
  }
}
```

실제 값은 본인 버킷·경로·프로파일에 맞게 바꾼다. 이 저장소 `main.tf`에는 아직 `backend`가 없으므로, 강의 순서대로 **버킷 생성 후** 위 블록을 추가하면 된다.

---

## 작업 순서 (슬라이드 요약)

1. **S3 버킷 생성** (버전 관리·암호화·퍼블릭 액세스 차단 등은 계정 정책에 맞게)
2. **`terraform` 블록에 `backend "s3"` 추가** (또는 기존 로컬 백엔드에서 S3로 변경)
3. **백엔드 변경 후에는 `terraform init`을 다시 실행**해야 한다.  
   기존 로컬 `tfstate`를 S3로 옮길 때는 프롬프트에 따라 **마이그레이션**을 선택한다.

```bash
cd /Users/kimtaeyoun/Desktop/Dev/udemy_terraform/terraform
terraform init
```

이미 상태가 있으면 Terraform이 **상태를 S3로 복사할지** 물을 수 있다. 팀 규칙에 맞게 진행한다.

---

## 정리

- 다중 사용자·CI에서는 **로컬 단독 tfstate**를 피하고 **원격 상태**를 쓴다.
- S3 백엔드는 **`terraform { backend "s3" { ... } }`**로 선언하고, 변경 시 **`terraform init` 재실행**이 필수다.
- 운영에서는 **상태 전용 계정 + 최소 권한 + 버전 관리**(및 잠금 테이블)까지 고려하는 것이 좋다.
