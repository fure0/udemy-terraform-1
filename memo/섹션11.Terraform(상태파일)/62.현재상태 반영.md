# 62. 현재 상태 반영

콘솔·CLI로 **클라우드 리소스를 직접 수정**하면(예: EC2에 태그 추가), **실제 인프라**는 바뀌는데 **`main.tf`는 그대로**이고, **`tfstate`도 아직 옛 값**일 수 있다.  
슬라이드 표현: **클라우드에서 손으로 고치면 코드와 tfstate 둘 다와 어긋난다**(드리프트).

반대로, **클라우드에 있는 사실을 상태 파일에 맞추고 싶을 때**가 “현재 상태를 상태에 반영”하는 흐름이다(슬라이드: 클라우드 위의 상태를 반영할 필요가 있다).

---

## `terraform refresh`

```bash
terraform refresh
```

| 인자 | 설명 |
|------|------|
| (기본) | 추가 인자 없이, 현재 워크스페이스의 상태를 대상으로 동작 |

**역할:** API로 **실제 리소스의 현재 속성**을 읽어와 **`tfstate`를 그에 맞게 갱신**한다. 코드(`main.tf`)는 자동으로 바뀌지 않는다.

최근 Terraform에서는 **`terraform plan` / `apply` 전에 refresh가 함께 실행**되는 것이 기본에 가깝다. 그래도 강의·문서에서는 **`refresh`만 따로 실행**해 “상태만 클라우드에 맞춘다”는 개념을 배우는 경우가 많다.

---

## 설정 차이(드리프트) 다루기 예시 (강의 슬라이드)

**목표:** 콘솔에서 만든 변경을 인지한 뒤, **코드가 정의한 모습으로 되돌리기**.

1. **EC2에 태그를 수동 추가** (예: `Message: Hello World`) — Terraform 밖의 변경.
2. **`terraform refresh`** — `tfstate`를 **클라우드의 현재 값**으로 맞춘다. 이제 상태에는 수동 태그가 반영된 것처럼 보일 수 있다.
3. **`terraform apply`** — `main.tf`에는 그 태그가 없으면, Terraform은 **코드가 원하는 상태**로 맞추기 위해 **수동으로 붙인 태그를 제거**하는 등의 변경을 한다(슬라이드: 코드에 맞춰 EC2 상태를 되돌림).

즉, **refresh는 “사실을 상태에 기록”**, **apply는 “코드가 말하는 이상 상태로 인프라를 맞춤”**에 가깝다.

---

## 정리

- **드리프트**: 코드·상태·실제 중 어느 한쪽만 바뀐 상태.
- **`terraform refresh`**: **실제 → tfstate** 방향으로 동기화(코드는 수동 수정 필요).
- 수동 변경을 **남기려면** 코드도 수정하고, **버리려면** `apply`로 코드 기준으로 되돌린다.
