# 60. 관리 밖 리소스 넣기

콘솔·CLI 등으로 **이미 만들어 둔 EC2**가 있는데, Terraform 상태(`tfstate`)에는 없으면 Terraform 입장에서는 그 인스턴스는 **관리 대상이 아니다**.  
이때 `main.tf`에 `aws_instance` 블록만 추가해도, 상태에는 여전히 해당 리소스가 없어서 **코드와 tfstate가 어긋난다**(슬라이드: 코드만 추가하면 tfstate와 차이가 난다).

**실존 리소스를 상태에 연결**해 두어야 이후 `plan` / `apply`가 그 인스턴스를 **같은 객체**로 다룬다. 그 작업이 **`terraform import`**다(슬라이드: tfstate에도 리소스를 가져온다).

---

## 명령

```bash
terraform import <ADDRESS> <ID>
```

| 인자 | 설명 |
|------|------|
| `<ADDRESS>` | 가져온 뒤 상태에 둘 **코드상 리소스 주소**(예: `aws_instance.manual`) |
| `<ID>` | 가져오려는 **실제 리소스 ID**(리소스 타입마다 형식이 다름) |

예: EC2는 보통 **인스턴스 ID**(`i-0abc123...`)를 쓴다.

```bash
terraform import aws_instance.manual i-0123456789abcdef0
```

---

## 전제

- **`import`는 `.tf` 코드를 자동 생성하지 않는다.**  
  먼저 **실제와 최대한 동일한** `resource` 블록을 작성해 두고, 그 주소로 가져온다.
- 가져온 직후 `terraform plan`을 돌리면 **속성 차이(drift)**가 남을 수 있다. 코드를 실제 설정에 맞게 다듬어 **plan이 안전한 수준**이 될 때까지 조정한다.

---

## 강의 흐름: 이미 만든 EC2 가져오기

1. **수동으로 EC2 기동** (콘솔 등)
2. **그 EC2에 대응하는 Terraform 코드 작성** (`ami`, `instance_type`, `subnet_id`, SG 등)
3. **`terraform import <ADDRESS> <인스턴스ID>`** 로 `tfstate`에 반영

이후에는 해당 주소로 Terraform이 그 EC2를 추적한다.

---

## `state mv`와의 차이

| 명령 | 용도 |
|------|------|
| `terraform state mv` | **이미 상태에 있는** 항목의 **주소(이름)만** 바꿀 때 |
| `terraform import` | **상태에 없는** 실존 리소스를 **처음으로** 상태에 넣을 때 |

---

## 정리

- 관리 밖 리소스는 **코드 + import**로 상태까지 맞춘다.
- import 이후에는 반드시 **`terraform plan`**으로 남은 차이를 확인한다.
- Terraform 1.5+에는 설정 기반 import(`import` 블록)도 있으나, 강의·CLI 중심이면 위 명령으로 충분하다.
