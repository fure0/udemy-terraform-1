# 61. 리소스 삭제

Terraform에서 **삭제**라고 하면 두 가지가 자주 섞인다.

1. **설정에서 빼고 `apply`**: Terraform이 **실제 클라우드 리소스까지** 없애려 한다(일반적인 `destroy` 방향의 삭제).
2. **상태에서만 제거**: **`terraform state rm`** — **tfstate에서만** 항목을 지우고, **AWS의 EC2 등은 그대로** 둔다. 즉 **관리 대상에서만 제외**한다(슬라이드: 어떤 리소스를 관리 대상 밖으로 / tfstate에서 리소스 삭제).

---

## 코드만 지우면 생기는 일

`.tf`에서 리소스 블록만 지우면 **코드와 tfstate가 어긋난다**(슬라이드: 코드만 삭제하면 tfstate와 차이가 난다).

그 상태에서 **`terraform apply`**를 하면, Terraform은 **원하는 상태(코드)에 맞추기 위해** 상태·클라우드에 남아 있는 리소스를 **삭제**하는 플랜을 낼 수 있다(슬라이드: tfstate와 차이가 있는 상태로 apply 하면 리소스가 사라진다).

정리하면, **코드에서 제거한 뒤 apply**는 곧 **실인프라 삭제**로 이어질 수 있으므로, `plan`을 반드시 확인한다.

---

## `terraform state rm`: 관리 대상에서 빼기

```bash
terraform state rm <ADDRESS>
```

| 인자 | 설명 |
|------|------|
| `<ADDRESS>` | **관리 대상에서 빼고 싶은** 리소스 주소(`terraform state list`로 확인) |

- **상태 파일에서 해당 주소만 삭제**한다.
- **EC2·RDS 등 실제 리소스는 삭제하지 않는다.** 콘솔에서 그대로 남는다.
- 이후 Terraform은 그 리소스를 **모른다**. 다시 쓰려면 `import` 등으로 다시 연결해야 한다.

예:

```bash
terraform state rm aws_instance.manual
```

---

## import 해 둔 수동 EC2를 없앨 때 (강의 흐름)

**「Terraform 손에서 떼고, 클라우드에서도 지우기」** 순서다.

1. **가져온 EC2를 관리 대상 밖으로**
   1. **tfstate에서 삭제** — `terraform state rm aws_instance.<이름>`
   2. **`.tf` 코드에서 해당 `resource` 블록 삭제**
2. **더 이상 필요 없는 수동 생성 EC2를 AWS에서 삭제** — 콘솔·CLI로 인스턴스 종료(이때는 Terraform이 관여하지 않음)

`state rm`을 먼저 하면, 코드를 지울 때 **의도치 않은 destroy 플랜**과 구분하기 쉽다. (팀 규칙에 따라 순서를 바꾸는 경우도 있으나, 강의 슬라이드는 위 순서다.)

---

## `destroy` / 코드 제거 `apply`와 비교

| 목적 | 대표적인 방법 |
|------|----------------|
| Terraform이 관리하던 리소스를 **클라우드에서도** 없앰 | 블록 삭제 후 `terraform apply`, 또는 `terraform destroy` / `-target` |
| **클라우드는 유지**하고 Terraform만 추적 중단 | `terraform state rm` 후 코드 정리 등 |

---

## 정리

- **실제 삭제**와 **상태에서만 삭제(state rm)**는 다르다.
- 코드·상태·클라우드 세 곳의 관계를 보고, **원하는 결과가 무엇인지**에 맞게 명령을 고른다.
- 수동 EC2 import 실습을 되돌릴 때는 **state rm → 코드 삭제 → 콘솔에서 EC2 삭제** 흐름이 슬라이드와 같다.
