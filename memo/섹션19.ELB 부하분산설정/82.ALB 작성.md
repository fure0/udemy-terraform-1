# 82. ALB 작성

**Application Load Balancer(ALB)** 를 Terraform **`aws_lb`** 로 만든다.  
섹션19는 ELB로 **AP 서버(`app_server`) 앞에 부하 분산·HTTP 진입점**을 두는 흐름이고, **82 → 83 타겟 그룹 → 84 리스너 → attachment** 순으로 이어진다.

다음: **83 타겟 그룹** · **84 리스너**

---

## ELB 데이터 구조 (슬라이드)

참조 화살표: **참조 원 → 참조 대상** (73번과 동일). Terraform은 **참조 대상을 먼저** 만든다.

```
로드 밸런서 (aws_lb)
    ↑
리스너 (aws_lb_listener) ──→ 타겟 그룹 (aws_lb_target_group)
                                    ↑
              타겟 그룹 attachment ──┘
                    ↑
              EC2 인스턴스 (aws_instance)
```

| 리소스 | 참조 대상 | 코드에서의 연결(예) |
|--------|-----------|---------------------|
| **리스너** | 로드 밸런서 | `load_balancer_arn = aws_lb.xxx.arn` |
| **리스너** | 타겟 그룹 | `target_group_arn = aws_lb_target_group.xxx.arn` |
| **attachment** | 타겟 그룹 | `target_group_arn = ...` |
| **attachment** | EC2 | `target_id = aws_instance.app_server.id` |

**82번은 맨 위 `aws_lb`(ALB)만** 작성한다. 리스너·TG·등록은 83~84.

---

## `aws_lb` 주요 인자 (슬라이드)

| 항목 | 타입 | 설명 |
|------|------|------|
| **`name`** | string | 로드 밸런서 이름 |
| **`internal`** | bool | **내부용 LB** 여부 (`true` = VPC 내부만, `false` = 인터넷 facing) |
| **`load_balancer_type`** | enum | `"application"` · `"network"` · `"gateway"` — **ALB는 `application`** |
| **`security_groups`** | string[] | LB에 붙일 **보안 그룹 ID** 목록 |
| **`subnets`** | string[] | LB가 사용할 **서브넷 ID** (AZ별 **2개 이상** 권장) |
| **`tags`** | map | 태그 |

ALB는 **L7(HTTP/HTTPS)** 용이다. 이 강의 웹 프론트(`web_sg` 80/443)와 맞는다.

---

## 이 프로젝트에 맞는 설계

| 항목 | 이 저장소에서 쓸 값 | 이유 |
|------|---------------------|------|
| **`load_balancer_type`** | `"application"` | HTTP로 AP(3000) 앞단 |
| **`internal`** | `false` | 인터넷에서 80/443 접근 (`web_sg`가 `0.0.0.0/0`) |
| **`security_groups`** | `[aws_security_group.web_sg.id]` | `security_group.tf`의 **web-sg** (80·443 인바운드) |
| **`subnets`** | 퍼블릭 서브넷 2AZ | `public_subnet_1a`, `public_subnet_1c` — ALB는 **서브넷당 ENI** |

`web_sg` → `app_sg` 3000 규칙은 이미 있으므로, ALB가 **web_sg**를 쓰면 트래픽이 AP EC2로 이어질 수 있다(83·84에서 TG·리스너로 연결).

---

## HCL 예시

`elb.tf` 등 새 파일에 두는 예:

```hcl
resource "aws_lb" "app" {
  name               = "${var.project}-${var.environment}-app-alb"
  internal           = false
  load_balancer_type = "application"
  security_groups    = [aws_security_group.web_sg.id]
  subnets = [
    aws_subnet.public_subnet_1a.id,
    aws_subnet.public_subnet_1c.id,
  ]

  tags = {
    Name    = "${var.project}-${var.environment}-app-alb"
    Project = var.project
    Env     = var.environment
  }
}
```

| 인자 | 이 예에서 |
|------|-----------|
| `name` | `tastylog-dev-app-alb` 형태 (`var.project`, `var.environment`) |
| `internal` | `false` — 퍼블릭 ALB |
| `security_groups` | **web_sg** 하나 |
| `subnets` | **1a·1c 퍼블릭** — 멀티 AZ |

## 의존 관계 (암시적)

`aws_lb` 블록이 아래를 **속성으로 참조**하므로 **추가 `depends_on` 없이** 순서가 잡힌다.

- `aws_vpc.vpc` (SG·서브넷 경유)
- `aws_security_group.web_sg`
- `aws_subnet.public_subnet_1a` / `public_subnet_1c`

77~79번 `terraform graph`로 보면 **서브넷·SG → LB** 방향이 보인다.

## apply 전 확인

```bash
cd terraform
terraform plan
```

- **add**: `aws_lb.app` (리소스 이름은 강의·파일에 맞게)
- **에러 예**: 서브넷이 한 AZ만 있으면 ALB 생성 정책에 걸릴 수 있음 → **퍼블릭 2개** 유지

현재 저장소에는 **`aws_lb` 코드가 아직 없음** — 강의 실습에서 `elb.tf` 등으로 추가.

## 83·84와의 역할 분담

| 강의 | 리소스 | 역할 |
|------|--------|------|
| **82 (지금)** | `aws_lb` | ALB **본체** (ENI·SG·서브넷) |
| **83** | `aws_lb_target_group` | 트래픽을 **어느 포트·어느 EC2 풀**로 보낼지 |
| **84** | `aws_lb_listener` | **80/443 수신** → 타겟 그룹으로 포워딩 |
| (attachment) | `aws_lb_target_group_attachment` | **EC2를 TG에 등록** |

ALB만 만들고 **리스너·TG가 없으면** URL로 접속해도 응답이 없다. 82는 **첫 단계**다.

---

## 정리

- **ALB** = `aws_lb` + `load_balancer_type = "application"`.
- **인터넷 facing**: `internal = false`, **web_sg**, **퍼블릭 서브넷 2AZ**.
- ELB 구성은 **LB → 리스너 ↔ 타겟 그룹 ← attachment ← EC2** 참조 구조.
